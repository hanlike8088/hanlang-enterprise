## 四、Phase 0 — 系统管理后台（第1-3周）

> **核心思想：** 让管理员可以在后台自定义配置，而不是每次调整都要改代码。
> 这样短期->中期->长期过渡时，只需要改配置，不重写系统。

### 0.1 为什么需要管理后台

| 如果硬编码 | 如果可配置 |
|-----------|-----------|
| 销售部重组了 -> 叫程序员改代码 | 管理员进后台修改组织架构 |
| 新增审批环节 -> 改代码 | 管理员配置新的审批节点 |
| 编号规则变了 -> 改代码 | 管理员改编码规则 |
| 短期转中期需调整岗位 -> 改代码 | 管理员调整配置，数据结构不变 |
| 长期按ISO体系定岗 -> 重写权限模块 | 管理员重新配置角色权限 |

### 0.2 管理后台包含的内容

> 这个后台不是给普通员工用的，是给系统管理员/IT人员用的。

#### 1) 组织管理

| 功能 | 说明 |
|------|------|
| 新增/编辑/删除组织 | 公司 -> 中心 -> 部门 -> 组 |
| 组织树形展示 | 一眼看清公司架构 |
| 短期 | 按金蝶现有组织架构配置 |
| 中期 | 按实际发展调整或体系认证需求新增 |

#### 2) 岗位管理

| 功能 | 说明 |
|------|------|
| 新增/编辑/删除岗位 | 打样技术员、销售员、研发主管... |
| 岗位归属组织 | 每个岗位属于哪个中心/部门 |
| 短期到中期 | 岗位定义可以逐步调整，不删数据 |

#### 3) 员工管理

| 功能 | 说明 |
|------|------|
| 员工花名册 | 姓名、工号、手机、邮箱 |
| 员工-岗位分配 | 一人一岗或一人多岗 |
| 启用/禁用 | 离职员工禁用，不影响历史数据 |

#### 4) 角色与权限管理

| 功能 | 说明 |
|------|------|
| 定义角色 | 如"打样技术员角色"、"销售员角色" |
| 权限分配 | 每个角色能访问哪些页面、执行哪些操作 |
| 角色-岗位绑定 | 一个岗位对应一个角色 |
| 示例 | 岗位"打样技术员" -> 可操作：查看任务/更新进度 |

#### 5) 编码规则配置

每个业务单据的编号格式由管理员配置，不是写死在代码里：

| 单据类型 | 格式模板 | 示例 |
|---------|---------|------|
| 打样工单 | {前缀}-{年份}-{流水号} | DY-2026-0001 |
| 报价单 | BJ-{年份}-{流水号} | BJ-2026-0001 |
| 销售订单 | SO-{年份}-{流水号} | SO-2026-0001 |
| 客诉单 | KS-{年份}-{流水号} | KS-2026-0001 |

**规则可自定义：** 前缀、年份位数、流水号位数、分隔符都可以在后台配。

#### 6) 状态流转配置

每个模块的单据在哪些状态之间怎么流转、谁能操作，由配置决定。

举例--打样工单的状态流转：

待审批 -> (研发主管通过) -> 已审批 -> (打样主管分配) -> 进行中 -> (打样技术员完成) -> 已完成
进行中 -> (异常上报) -> 已暂停 -> (问题解决) -> 进行中

如果未来流程变了（比如中间加一个"技术评审"环节），管理员在后台配置里添加一个状态和流转关系就行，不用改代码。

#### 7) 系统设置

| 设置项 | 说明 |
|--------|------|
| 金蝶对接配置 | 服务器地址、账套、用户名（已在.env）|
| 系统边界 | 哪些数据从金蝶只读、哪些可在本系统写（后面细化）|
| 通知方式 | 系统内通知/邮件（将来）|

### 0.3 管理后台的界面

不需要很漂亮，功能完整就行。侧边栏多一个"系统管理"菜单：

```
系统管理
 |-- 组织管理
 |-- 岗位管理
 |-- 员工管理
 |-- 角色权限
 |-- 编码规则
 |-- 状态流转
 |-- 系统设置
```

### 0.4 实施方式

**开发周期**（3周）：

**第1周：组织+岗位+员工**
- 数据库设计：组织树、岗位、员工表
- CRUD API + 前端页面
- 树形组织展示

**第2周：角色+权限**
- 角色CRUD
- 权限定义（页面级+操作级）
- 角色-权限分配 + 岗位-角色绑定
- 权限检查中间件

**第3周：编码规则+集成+备份**
- 编码规则配置
- 与金蝶对接集成测试
- 自动备份机制
- 管理员使用说明文档

**产出：** 系统管理后台上线，组织-岗位-员工-角色-权限全链条可配置。
- 组织/岗位/员工的CRUD
- 角色权限的基本分配
- 编码规则配置

**后续各Phase**：
- 状态流转 -> 按模块逐个配置
- 系统设置 -> 随需求增加

> 这样设计，当公司从短期->中期->长期过渡时，管理员在后台上调整配置，数据无缝衔接，不重构系统。

## 一、整体开发逻辑

### 为什么要先做研发、再做销售？

~~~mermaid
flowchart LR
    RD[技术研发] -->|1.管好产品/BOM/图纸| SM[销售]
    SM -->|2.报价时调用产品数据| SC[供应链]
    SC -->|3.采购物料| MF[生产]
    MF -->|4.消耗物料完成订单| FN[财务]
    FN -->|5.算成本分析利润| MG[管理层看板]
~~~

**数据依赖链：** 销售报价需要产品数据和BOM成本 → 采购需要销售订单 → 生产需要采购到料 → 财务需要生产数据

所以：**先做研发**把产品数据底座建好，**再做销售**直接调用底座，数据不会断。

## 二、开发协作规则

> 这些规则决定了我们怎么配合、怎么验收、怎么让系统真用起来。

### 3.1 迭代方式——小步快跑，不搞大爆炸

每个模块不一次做完整再给你看，而是拆成多个功能点，逐个交付：

| 阶段 | 做法 |
|------|------|
| 拆 | 一个模块拆成3-5个功能点（如打样工单=提交→审批→分配→执行→查看）|
| 做 | 做一个功能点，一般1-2天 |
| 给你试 | 我叫你"XX好了，来试试" |
| 你验收 | 你跟着验收步骤走一遍 |
| 确认或改 | 你说"可以"或"这里要改" |
| 下一个 | 没问题了就做下一个功能点 |

这样你不会等了两周才看到结果，我也能及时发现方向是否偏了。

### 3.2 验收流程——你怎么知道"做完了"

每个功能点交付时，我会附一份验收清单：

**示例——打样申请的验收清单：**

| # | 步骤 | 期望结果 | 你打勾 |
|:-:|------|---------|:-----:|
| 1 | 打开系统，用销售员账号登录 | 看到首页和菜单 | ☐ |
| 2 | 点"打样管理"→"新建打样申请" | 弹出申请表单 | ☐ |
| 3 | 填写产品名称、数量、日期、说明，上传一张图纸 | 表单提交成功 | ☐ |
| 4 | 提交后，在列表里看到刚才的申请 | 列表中多了一条记录 | ☐ |
| 5 | 用研发主管账号登录，看到待审批列表 | 看到刚才的申请 | ☐ |
| 6 | 点击"通过" | 状态变为"已审批" | ☐ |

你跟着走一遍，全部打勾 = 这个功能点验收通过。有一项走不通或跟你想的不一样 = 我改到你满意。

### 3.3 变更管理——怎么让系统真用起来

功能做出来不等于有人用。这步做不好，系统就是摆设：

| 阶段 | 做法 | 时间 |
|------|------|:----:|
| 试点 | 找一个配合度高的部门先试（比如先让一个销售员试打样申请）| 模块做完后第1周 |
| 收集反馈 | 试点期间记录：哪里不好用、哪里缺功能、哪里太复杂 | 第1-2周 |
| 调整 | 根据反馈修改，调到试点人员说"还行，比微信发来发去好"| 第2-3周 |
| 推广 | 调稳了再扩大到整个中心 | 第3-4周 |
| 正式用 | 旧流程（纸质/Excel）停用，新系统成为唯一通道 | 第4周起 |

**关键原则：** 好用自然有人用。不是靠行政命令推，而是让系统比旧的干活方式更方便。

### 3.4 数据迁移——第一批数据从哪来

| 数据类型 | 来源 | 处理方式 | 时间 |
|---------|------|---------|:----:|
| 组织/岗位/员工 | 金蝶现有组织架构 + 你提供 | Phase 0时按金蝶配好 | Phase 0 |
| 客户数据 | 金蝶（已对接可读取）+ 销售员现有Excel | Phase 1.2 做CRM时导入 | Phase 1.2 |
| 历史打样记录 | 如果有纸质/Excel记录 | 可以导入做参考，但强不强制 | Phase 1.1 |
| 现有图纸 | 研发人员电脑里 | 用系统重新上传，旧图纸作为参考 | Phase 1.1 |
| 金蝶物料/供应商 | 已对接，实时读取 | 直接显示，不需要手动录入 | 已有 |

> **原则：** 系统上线后产生的新数据必须进系统。历史数据能导入就导入，不能导入的逐步放弃不影响业务。

## 五、Phase 1.1 — 技术研发先行（4周）

### Module 1：技术打样工单系统（第1-2周）

**一句话：** 把"发微信/打电话催打样"变成系统里可追踪的任务

**需求定义——做完了你能：**
1. 销售员（需求人员）在系统里提交一份"技术打样申请"，填写：客户需求产品、数量、要求完成日期、打样说明、参考图纸（附件上传）
2. 研发主管在线审批打样申请（通过/退回/修改意见）
3. 打样主管看到已审批的申请，指派给具体打样人员，设定目标完成日
4. 打样技术员在系统里收到任务，可以更新进度：开始处理、进行中、异常暂停（填写原因）、已完成
5. 销售员（需求人员）可以随时查看自己申请的打样到了哪一步
6. 系统自动统计：打样平均完成天数、各打样技术员工作量、超期打样预警

**验收标准（怎样算"做完了"）：**
- ✅ 销售员提交一份打样申请，上传了客户参考图纸
- ✅ 研发主管在系统里看到了待审批列表，点击通过了申请
- ✅ 打样主管把任务指派给了张三（打样技术员），张三登录看到了任务
- ✅ 张三更新进度为"已完成"，销售员页面上状态变了

**依赖关系：** 需要当前NPI项目管理模块中的"人员/组织"数据

---

### Module 2：图纸版本管理（第2-3周）

**一句话：** 不再出现"图纸版本搞错了、用的是旧版"

**需求定义——做完了你能：**
1. 研发工程师上传图纸（支持PDF/DWG），填写：所属产品、图纸名称、图号、版本号
2. 系统自动记录每次上传为新版本，旧版本保留可查看（不能删除）
3. 查看图纸时清楚显示当前最新版本号、上传人、上传时间
4. 可以比较两个版本的差异（上传时间、修改说明）
5. 图纸关联到产品，看产品资料时能看到所有关联图纸
6. 图纸被引用时（如在打样工单中）显示当前版本号

**验收标准：**
- ✅ 上传一份电机总装图PDF，填写产品YJ4820-0028，图号DWG-001，版本V1.0
- ✅ 再次上传同一图号，系统自动记为V1.1，旧版V1.0仍可查看
- ✅ 在产品YJ4820-0028详情页能看到这份图纸

**依赖关系：** 需要产品主数据（Module 3）先定义好产品列表

---

### Module 3：产品主数据与设计BOM（第3-4周）

**一句话：** 公司每款电机在系统里有完整档案，不再散落在个人电脑里

**需求定义——做完了你能：**
1. 研发工程师创建产品档案：产品编码、产品名称、型号、规格参数（电压/功率/转速/转矩等）、产品分类（循环扇电机/罩极电机/无刷直流电机等）
2. 可以为产品创建BOM（物料清单）：物料编码、物料名称、用量、单位、层级
3. BOM支持多层结构（产品→组件→零件），可以逐层展开查看
4. 产品可以关联：图纸、专利文档、技术规范、检验标准
5. 产品列表支持搜索/筛选（按分类、按状态）
6. 产品有生命周期状态：开发中→试产→量产→停产

**验收标准：**
- ✅ 创建产品"YJ4820-0028静音电机"，填入完整规格参数
- ✅ 为它创建BOM：机壳×1、定子组件×1、转子组件×1、轴承×2、端盖×1
- ✅ 把之前上传的图纸关联到这个产品
- ✅ 在列表中搜索"4820"能搜到这个产品

**依赖关系：** 无（新模块，从零建）

### Module 4：专利关联产品（第3-4周，与Module 3并行）

**一句话：** 公司33份专利不再是散落的PDF，而是跟产品关联的"技术资产"

**需求定义——做完了你能：**

**验收标准：**
- ✅ 33份专利已导入且可查阅（已有）
- ✅ 把"一种全自动铰孔压轴机"专利关联到YJ4820-0028产品
- ✅ 在产品YJ4820-0028页面上能看到这个专利

**依赖关系：** 需要Module 3的产品主数据

## 六、Phase 1.2 — 销售管理跟进（第5-8周）

### Module 5：CRM客户管理（第5-6周）

**需求定义：**
1. 创建客户档案：客户名称、联系人、电话、地址、信用额度、账期

**验收标准：** ✅ 新建一个客户"美的电器" → 录入联系人 → 查看客户360°视图

### Module 6：快速报价系统（第6-7周）

**一句话：** 不用翻Excel算成本，系统自动算

**需求定义：**
1. 选择产品→系统自动带入BOM物料清单和数量
2. 填入各物料当前采购价（从金蝶读取参考价）
3. 系统自动计算：材料成本 + 人工估算 + 制造费用 = 报价参考
4. 手动调整最终报价，填入利润率
5. 生成报价单PDF，可直接发给客户
6. 历史报价可查，同类产品快速复用

**验收标准：** ✅ 为YJ4820-0028创建一个报价 → 系统自动带出BOM → 填入价格 → 生成PDF

**依赖关系：** 需要Phase 1.1的产品BOM数据

### Module 7：销售订单跟踪（第7-8周）

**一句话：** 不再"客户问一下进度，你再去问一圈"

2. 订单状态机：待确认→已确认→生产中→已发货→已签收→已完成
3. 订单关联客户、产品、金额、交期
4. 交期预警：离交期还有N天时自动提醒

## 七、验收总原则

> 每个模块做完后，问三个问题来判断它是否合格：

| # | 问题 | 验证人 |
|---|------|--------|
| 1 | **你自己能不能在系统里把这件事完整跑一遍？**（不依赖别人帮你操作）| 该模块的负责人 |
| 2 | **数据对不对？**（比如打样申请单上的信息跟纸质单对比）| 你 |
| 3 | **如果明天就让你们部门用这个系统干活，会不会卡住？**| 实际用户 |

**什么叫"做完了"：**
- 你亲自验收通过了（不是我说做完了就做完了）
- 你觉得"嗯，能用，比我现在的Excel好"

## 八、关于中长期的说明

### 8.1 中长期的岗位定义方式

短期和中期按照金蝶现有组织架构和岗位来定。

中长期（1-3年）则按体系认证的要求来定岗——如ISO 9001质量管理体系、IATF 16949汽车行业体系等。这些体系标准对每个岗位的职责、权限、能力有明确定义。

**定岗可以不定人：** 职位（如"品质主管"）按体系标准定义好职责和KPI，但具体是谁担任可以根据实际情况调配。

## 十、Phase 1.3 — 供应链中心（第9-12周）

> 核心思想：供应商从报价到付款的全链条监管，不再靠电话和微信催

### Module 1：供应商全生命周期管理（第9周）

**一句话：** 每个供应商有完整档案、有绩效评分、有淘汰机制

**需求定义——做完了你能：**
1. 创建供应商档案：编码、名称、联系人、电话、地址、银行账户、税号、供货类别
2. 供应商分类：原材料供应商/外协件供应商/辅料供应商/设备供应商
3. 供应商准入流程：申请→资质审核→样品确认→批准→纳入合格供应商名录
4. 供应商绩效评分（QCDS）：质量得分（来料合格率）+ 成本得分（价格竞争力）+ 交付得分（准时率）+ 服务得分（响应速度）
5. 绩效看板：每个供应商的评分趋势图，按季度自动生成
6. 供应商分级：A级（优秀）→ B级（良好）→ C级（观察）→ D级（淘汰）
7. D级供应商自动预警，触发淘汰审批流程

**验收标准：**
- ✅ 新建一个供应商"慈溪市忱科塑料厂"，填入完整信息
- ✅ 录入一批来料检验数据，系统自动更新该供应商的质量评分
- ✅ 绩效看板上看到该供应商的评分趋势图表

---

### Module 2：采购订单管理（第10周）

**一句话：** 从下单到到货全程透明，不用再打电话催

**需求定义——做完了你能：**
1. 创建采购订单：选择供应商、选择物料、填写数量/单价/交期、备注
2. 采购订单状态机：草稿→已确认→供应商确认→已发货→已到货→检验中→已入库→已关闭
3. 订单进度可视化：当前处于哪个状态、每个状态的时间戳
4. 交期预警：离交期还有3天时自动标黄、超期标红
5. 采购订单关联：一份订单可以关联到多个销售订单（知道这批料是给哪个客户订单买的）
6. 支持分批到货：一次采购可以分多次到货、多次检验、多次入库

**验收标准：**
- ✅ 创建一笔采购订单，给供应商"宁波亿利通"采购电线电缆100米
- ✅ 订单状态从"已确认"走到"已到货"
- ✅ 将该订单关联到一个销售订单
- ✅ 系统显示交期预警

---

### Module 3：仓库管理（第11周）

**一句话：** 扫码出入库，库存数据实时准确

**需求定义——做完了你能：**
1. 仓库/库位设置：多个仓库（原材料仓、半成品仓、成品仓）、每个仓库分库区库位
2. 入库操作：采购到货→扫码→上架→库存增加
3. 出库操作：生产领料/销售发货→扫码→下架→库存减少
4. 即时库存查询：按仓库、按物料、按类别实时查询当前库存量
5. 库存流水：每一笔出入库都有记录（时间、物料、数量、操作人、单据号）
6. 库存预警：低于安全库存时自动预警
7. ABC分类管理：A类物料重点管控（高频盘点）、C类物料简化管理

**验收标准：**
- ✅ 设置三个仓库：原材料仓、半成品仓、成品仓
- ✅ 做一笔入库：物料"漆包线"入库100卷
- ✅ 查询即时库存，看到漆包线库存为100卷
- ✅ 做一笔出库：生产领用漆包线10卷
- ✅ 库存流水里看到出入库两条记录

---

### Module 4：对账/发票/付款（第12周）

**一句话：** 三单匹配自动对账，差异自动预警

**需求定义——做完了你能：**
1. 采购对账：系统自动匹配 采购订单 + 入库单 + 供应商发票（三单匹配）
2. 匹配差异自动标识：数量差异、单价差异、金额差异
3. 财务人员审核匹配结果，确认无误后生成应付账款
4. 付款计划：根据供应商账期自动生成付款计划
5. 付款执行：记录实际付款日期、付款金额、付款方式
6. 应付账款账龄分析：超期未付自动预警
7. 供应商对账单：可按月度/季度生成对账单PDF，发给供应商确认
8. 与金蝶同步：应付数据确认后，写入金蝶（需confirmed确认）

**验收标准：**
- ✅ 一笔采购订单到货入库后，系统自动生成待对账记录
- ✅ 三单匹配成功（订单价=入库价=发票价），自动通过
- ✅ 三单匹配有差异（发票价与订单价不符），自动标红预警
- ✅ 财务确认后，生成应付账款记录
- ✅ 查看应付账款账龄分析表

## 十二、Phase 3 — 品质中心（第17-20周）

### Module 1：来料检验 IQC（第17周）

一句话：每批来料按标准检验，不合格品不会流入产线

需求定义：
1. 检验标准定义：按物料设定检验项目、规格上限下限、抽样方案（AQL）
2. 来料登记：采购到货后触发待检任务，扫码或手动登记
3. 检验录入：逐项录入测量值，系统自动判定合格/不合格
4. 不合格处置：让步接收 / 退货 / 降级使用
5. 退货触发：自动通知采购和供应商
6. 与采购单关联：检验结果关联到采购订单和入库单

验收标准：
- 定义一个来料检验标准（漆包线：线径、绝缘强度）
- 一笔采购到货后生成待检任务
- 录入检验数据，系统自动判定
- 不合格品触发退货流程

### Module 2：过程检验 IPQC（第18周）

一句话：首件确认+巡检，问题在产线内解决

需求定义：
1. 首件确认：每班次/每次换型时，操作员送首件→质检员检验→通过后量产
2. 巡检计划：按频率（每2小时）自动生成巡检任务
3. 巡检录入：记录巡检结果，异常时触发NCR
4. 工序质量控制：统计各工序的CPK

验收标准：
- 创建一个首件检验任务并完成检验
- 系统自动生成今日巡检任务
- 巡检发现异常，自动触发不合格品流程

### Module 3：出货检验 OQC + 不合格品管理 NCR（第19周）

一句话：成品出厂前把关，不合格品有闭环

需求定义：
1. 出货检验：成品完工→待检→检验→合格放行/不合格处置
2. NCR不合格品：标识→隔离→评审→处置（返工/报废/让步）
3. CAPA纠正预防：根本原因分析→措施→验证→关闭
4. 统计：客诉率PPM、质量成本

验收标准：
- 完成一次出货检验
- 创建NCR不合格品记录，走完评审处置流程
- 创建一个CAPA，从措施到验证关闭

### Module 4：量具/仪器管理（第20周）

验收标准：
- 录入量具台账，设定校准周期
- 校准到期自动提醒
- 记录校准结果


## 十三、体系集成架构（P1-P8）

> 这不是新功能，而是把现有18个模块从一个"功能集合"升级成"统一业务平台"

### P2：统一权限中间件

目标：Admin的RBAC模型扩展到所有模块

做法：
1. 创建一个 @RequirePermission(resource, action) 装饰器
2. 封装成全局 PermissionGuard
3. 为每个模块的 Controller 添加权限装饰器
4. 权限定义自动化：系统启动时自动扫描所有 Controller 生成权限清单

### P3：统一编码规则

目标：所有单据编号由 Admin CodingRule 统一管理

做法：
1. 所有模块的序号生成调用 AdminCodingRuleService
2. 覆盖：打样工单、报价单、订单、采购单、客诉单、NCR、CAPA

### P4：统一状态机

目标：所有单据的状态变化走统一状态机

做法：
1. 设计通用状态机接口（transition, guard, action）
2. 每个模块定义自己的状态图
3. 状态变更记录到审计日志

### P5：审批流引擎

做法：
1. 简单的审批流：根据金额/类型自动匹配审批人
2. 审批记录统一存储

### P6：模块事件总线

做法：
1. 用 NestJS EventEmitter 或简单的事件总线
2. 定义标准事件（order.confirmed, material.received 等）
3. 各模块监听/触发事件

### P7：跨模块看板

做法：
1. 财务看板：应收/应付/成本汇总
2. 订单全链路追踪：销售→采购→生产→发货

### P8：审计日志

做法：
1. 审计日志模型（谁/什么时间/什么动作/变化内容）
2. 全局审计拦截器自动记录
